## 1. 产品定位

PolicyFlow AI 前端是企业内部制度助手与轻量管理后台。它服务可信制度问答、引用溯源、知识维护和质量管理，不是通用聊天产品、OA、工单系统或低代码平台。

核心价值排序：

1. 回答有证据；
2. 权限不越界；
3. 状态可理解；
4. 操作可追踪；
5. 管理流程足够轻量；
6. 视觉效果服从业务可信度。

## 2. 用户角色

| 角色 | 核心目标 | 前端入口 |
|---|---|---|
| `employee` | 查询制度、查看引用、管理自己的草稿 | 问答、草稿、可读知识库 |
| `kb_admin` | 维护知识库、文档、FAQ 和评估 | 知识管理、FAQ、评估 |
| `sys_admin` | 用户、Skill、MCP、审计和全局管理 | 全部管理入口 |

资源级 `read/write/admin` 权限以知识库 API 返回为准，不能仅由全局角色推断。

## 3. MVP 业务闭环

### 3.1 当前可交付闭环

```text
登录 -> 恢复身份 -> sys_admin 查看用户 -> 创建用户 -> 修改角色
```

该闭环只依赖当前已实现后端接口，应优先完成。

### 3.2 目标业务闭环

```text
员工提问 -> 权限过滤后的检索 -> 答案 + 引用 -> 反馈/生成草稿
管理员上传文档 -> 索引 -> 员工可查询 -> FAQ/评估验证质量
系统管理员配置能力 -> 查看调用和审计结果
```

目标闭环必须跟随后端 API 就绪逐步开放。

## 4. MVP 包含

- 登录和身份恢复；
- 应用壳、导航、权限反馈；
- 用户列表、创建和角色修改；
- 制度问答、引用、无证据状态和反馈；
- 知识库与文档上传、索引状态；
- 草稿查看、编辑、确认、丢弃和导出；
- FAQ 审核；
- Skill、MCP、审计和 Eval 的轻量管理；
- 加载、空、错误、无权限、404、离线等基础状态。

## 5. 明确不做

- 完整审批流、工单流、OA 表单引擎；
- 可视化 Agent/Workflow 编排；
- 富文本多人协作；
- 多租户和白标；
- 用户自定义仪表盘；
- 真实邮件、日历或外部系统自动提交；
- 前端直接访问数据库、LightRAG、LLM 或 MCP；
- 浏览器保存模型密钥或后端管理密钥；
- 为尚未实现的后端功能制造“可用”假象。

## 6. 范围判断规则

新需求满足以下任一条件，应移出当前任务：

- 需要新增后端表或业务流程；
- 需要修改角色模型；
- 需要真实外部系统写操作；
- 需要引入新的全局状态体系；
- 需要一次改动三个以上 feature；
- 不属于当前 Phase 的完成定义。

处理方式：记录为“后端缺口”“架构决策”或“后续 Backlog”，不要顺手实现。

## 7. 本文完成条件

- 角色、范围和非目标清晰；
- 当前闭环与目标闭环分开；
- 新需求有明确归类规则；
- 未把任何设计态 API 描述成已实现。
